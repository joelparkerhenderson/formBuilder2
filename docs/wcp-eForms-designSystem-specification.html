WCP eForms design system specification

This document describes the design system common to the four patient forms: Waiting list card, Outpatient outcome, Operation note, and Treatment summary. It is derived from design.md and intentionally avoids form-specific field inventories.

1. System overview

The system provides high-contrast clinical electronic forms with editable views (EV), read-only views (RoV), version history, draft/final state, patient addressograph headers, and a consistent footer save/cancel/authentication workflow.

  • Purpose: capture structured clinical data using repeatable fbXxx components.
  • Principles: clarity, compact clinical scanning, visible state, robust persistence, and controlled version history.
  • Scope: web-based patient forms running inside formBuilder2 and SWAS static deployments.
  • Accessibility target: WCAG 2.2 AA, with native controls preferred and ARIA added where custom widgets require it.

Typography uses Google Roboto across controls, labels, headings, buttons, and layout text. Material Icons are excluded from the Roboto override so ligature icons render correctly. Addressographs use Arial/Helvetica to match printed-card style.

Standard input borders use 0.1rem solid silver and 0.4rem radius. Form shell, header, footer, and main workspace backgrounds are white.

3. Layout foundations

  • Every form sits inside fbLayout.
  • The top header contains fbHeader, form title, and fbAddressograph.
  • The footer uses fbBottomControlsRow and remains at the bottom of the viewport.
  • The left navigation panel is visible on desktop and hidden below 768px.
  • The central workspace uses full available panel width with padding 0.8rem 0 4rem 0.
  • Rows are created with fbGridRow; cells are created with fbGridCell.
  • Desktop labels are equalised within rows. Mobile layouts collapse to a single column and reset label heights.

4. Sections, rows, questions and subquestions

  • Forms are partitioned into multiple high-level container components fbSection.
  • Each fbSection container includes a dedicated section header, implemented via the fbSectionHeader component displaying a high-contrast visual title bar, followed by a list of child controls.
  • Inside each fbSection, the child content is structured as a list of fbGridRow containers, or sometimes just individual questions that are not grouped inside a row.
  • Row elements use the fbGridRow layout blocks to arrange questions horizontally using CSS grid.
  • Where there are four questions in a row, the fbGridRow contains exactly four fbGridCell components, each of which wraps a single question. If there is a second or third set of questions, there is a second or third separate fbGridCell nested inside the row.
  • Grid spaces and relative horizontal weights are divided cleanly using <fbGridCell span={n}> layout blocks.
  • Individual fields are wrapped in fbQuestion, which provides visual labels and structured padding.
  • Hidden display-conditional subquestions render recursively nested inline beneath selected parent triggers inside matching sections.

5. Behaviour patterns

Required fields

Required fields show a red asterisk. Save is disabled until required field completeness checks pass. Conditional subquestion values are reset when their parent value is cleared or changed.

Highlighting

Hover/focus highlighting alternates lighter and darker yellow across nested questions, groups, cells, and subquestions. The application-level layout owns the event wiring.

Tooltips

Custom tooltip bubbles appear near labels and controls, clamp within the viewport, and cascade from child controls to their containing question and section where helpful.

Save workflow

Forms save immutable versions. Draft saves require a draft confirmation popup; final saves require final state. Save feedback uses the shared popup family. A stale edit cannot create a new version if another user/window has already saved a newer version.

Read-only view

RoV omits empty details unless a grid placeholder is required, maps option values to labels, preserves multiline text, and renders coded/not-coded status indicators. Superseded versions show the Superseded badge and Back only.

Form state and change detection

Editable views keep user input in formState, with companion arrays for row lists where needed. The initial snapshot is compared with the current state to enable Save, prompt before RoV/Cancel transitions, and reject stale saves against newer persisted versions.

9. Behaviour patterns (detail)

Required fields

Required fields show a red asterisk. Save is disabled until required field completeness checks pass. Conditional subquestion values are reset when their parent value is cleared or changed.

  • Handled centrally by fbQuestion, fbTextInput, fbTextArea, and fbMSISelector.
  • If a label exists, the asterisk is grouped with the last word of the label in a nowrap wrapper so the asterisk does not wrap alone.
  • If there is no label, for example hidden nested subquestions, the asterisk is displayed immediately to the right of the control and top-aligned.
const RequiredMark = () => <span style={{ color: fbRed }}>*</span>;

function labelWithRequired(label, required) {
  if (!required) return label;
  const words = String(label).trim().split(/\s+/);
  const last = words.pop() || "";
  return <>{words.join(" ")}{" "}<span style={{ whiteSpace: "nowrap" }}>{last} <RequiredMark /></span></>;
}

Highlighting

Hover/focus highlighting alternates lighter and darker yellow across nested questions, groups, cells, and subquestions. The application-level layout owns the event wiring.

  • Outer question containers use fbActiveLighterYellow.
  • Inner active inputs, radios, check items, and subquestions use fbActiveDarkerYellow.
  • Deeply nested active items alternate between the two yellows. Table cells count as a nesting level so table-contained choices highlight correctly.
function highlightElement(target) {
  const activeNodes = target.closest(".fb-question-container, .fb-radio-checkbox-item, td");
  let depth = 0;
  for (let node = activeNodes; node; node = node.parentElement?.closest(".fb-question-container, .fb-radio-checkbox-item, td")) {
    node.style.backgroundColor = depth % 2 === 0 ? fbActiveLighterYellow : fbActiveDarkerYellow;
    depth += 1;
  }
}

Tooltips

Custom tooltip bubbles appear near labels and controls, clamp within the viewport, and cascade from child controls to their containing question and section where helpful.

  • fbToolTip renders absolute-positioned hover/focus help bubbles and avoids layout shift.
  • Manual close buttons use a solid black border and compact padding.
  • Grouped radio/check tooltips appear below the option label, but move above when an open subquestion would otherwise be obscured.
  • Open subquestions are detected with fb-subquestion-wrapper and fb-subquestion hooks.
function clampTooltip(left, width) {
  const margin = 8;
  return Math.max(margin, Math.min(left, window.innerWidth - width - margin));
}

function hasOpenSubquestion(el) {
  return Boolean(el.closest(".fb-radio-checkbox-item")?.querySelector(".fb-subquestion-wrapper, .fb-subquestion"));
}

Save workflow

Forms save immutable versions. Draft saves require a draft confirmation popup; final saves require final state. Save feedback uses the shared popup family. A stale edit cannot create a new version if another user/window has already saved a newer version.

  • Completeness is evaluated by form-specific areRequiredFieldsComplete() functions.
  • Disabled buttons use a solid silver background with white text.
  • Changed forms enable the save button with a green variant; unchanged forms leave it disabled.
  • fbSavingPopup, fbSavedPopup, fbSaveErrorPopup, fbStaleSavePopup, and fbDraftPopup communicate workflow state.
  • The footer password field clears two seconds after blur, but Save captures the current password immediately if clicked before that timeout expires.
async function handleSaveAndClose() {
  if (!areRequiredFieldsComplete(formState)) return;
  if (!finalChecked) {
    setPopup("draft");
    return;
  }
  if (formChanged && !password) {
    setPopup("password");
    return;
  }
  setPopup("saving");
  await saveClinicalVersion({ formState, finalChecked, password });
  setPopup("saved");
}

Read-only view

RoV omits empty details unless a grid placeholder is required, maps option values to labels, preserves multiline text, and renders coded/not-coded status indicators. Superseded versions show the Superseded badge and Back only.

  • Checked radio or checkbox values render as plain text strings with fontWeight: 500; unselected values are omitted.
  • Select values are mapped to complete descriptive labels.
  • Spacing grids are preserved by retaining empty grid wrappers where a column must not collapse.
  • RoV coded indicators align to the top of the first line of wrapped data.

Form state and change detection

Editable views keep user input in formState, with companion arrays for row lists where needed. The initial snapshot is compared with the current state to enable Save, prompt before RoV/Cancel transitions, and reject stale saves against newer persisted versions.

export const compareFormStatesObj = (snap, current) => {
  if (!snap || !current) return false;

  const allKeys = Array.from(new Set([
    ...Object.keys(snap.formState || {}),
    ...Object.keys(current.formState || {})
  ]));

  for (const key of allKeys) {
    const v1 = snap.formState?.[key] == null ? "" : snap.formState[key];
    const v2 = current.formState?.[key] == null ? "" : current.formState[key];
    if (String(v1) !== String(v2)) return false;
  }

  if (snap.procedures && current.procedures) {
    if (snap.procedures.length !== current.procedures.length) return false;
    for (let i = 0; i < snap.procedures.length; i++) {
      const p1 = snap.procedures[i];
      const p2 = current.procedures[i];
      if (p1.side !== p2.side || p1.procedure !== p2.procedure || p1.additionalInfo !== p2.additionalInfo) {
        return false;
      }
    }
  }

  if (JSON.stringify(snap.anticoagChecked) !== JSON.stringify(current.anticoagChecked)) return false;
  if (Boolean(snap.highlySensitive) !== Boolean(current.highlySensitive)) return false;
  if (Boolean(snap.finalChecked) !== Boolean(current.finalChecked)) return false;
  return true;
};

11. Other Algorithms

Enter key suppression

function preventAccidentalEnterSubmit(event) {
  const target = event.target;
  if (event.key !== "Enter") return;
  if (target instanceof HTMLTextAreaElement) return;
  if (target instanceof HTMLButtonElement && target.type === "submit") return;
  event.preventDefault();
}

Read-only value rendering

function displaySelectedOptions(options, selectedValues) {
  return options
    .filter((option) => selectedValues.includes(option.value))
    .map((option) => option.label);
}

SNOMED/Shades of Pale parser cleanup

function normalizeShadesOfPaleResponse(raw) {
  return raw
    .replace(/!n/g, "null")
    .replace(/!f/g, "false")
    .replace(/!t/g, "true");
}

Clinical date display

function formatClinicalDate(date) {
  if (!date) return "";
  return new Intl.DateTimeFormat("en-GB", { day: "2-digit", month: "short", year: "numeric" })
    .format(new Date(date))
    .replace(/ /g, "-");
}

13. Accessibility requirements

  • Prefer native controls before ARIA replacements.
  • Every input must have a programmatic label and visible label unless a compact clinical table cell makes visible context unambiguous.
  • Custom popups must expose dialog semantics and focus behaviour before being considered WCAG-complete.
  • Autocomplete controls must follow combobox/listbox semantics.
  • Status icons must have text equivalents.
  • Keyboard focus must remain visible and logical across EV, RoV, history, date picker, tooltip, and modal interactions.
