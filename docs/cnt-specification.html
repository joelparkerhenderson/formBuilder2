Case note tracker specification and user guide

Case note tracker is a React prototype for tracking paper case-note volumes, requests, pick lists, batches, tags, and custodial locations. It uses the formBuilder2 visual language and components so the tracker feels consistent with the clinical forms application. This guide describes how users complete the supported tasks and records the current implementation contract.

The shared eForms visual vocabulary is specified in WCP eForms design system specification. CNT-specific behaviour is maintained in cnt.spec.md and future server endpoints are listed in docs/cnt.restAPI.md.

User guide

Logging in and setting preferences

Open /formBuilder2/caseNoteTracker.html, choose a simulated user, then confirm Health board, Locality, and Facility preferences. These preferences prefill filters such as Select clinic(s) and workflow defaults such as Register volume locations.

Searching for a patient

From formBuilder2 Patient search or Patient registry, use the Case notes action to open the patient's Case notes locator. From the CNT home page, simulated NHS number and hospital number scan buttons open a random patient locator.

Opening a case notes locator

The locator groups volumes by Health board, Locality, Type, and Volume number. Use checkboxes to select volumes, expand/collapse group headings, and History to inspect a volume's movement events. Selected enclosing groups remain open until their selected volumes are cleared.

Requesting a volume or set of volumes

On Case notes locator or Case notes selector, tick one or more volumes and choose Request selected. The request workflow is only allowed when every selected volume is in a location with at least one custodian; otherwise an Error popup explains the problem. The Request page is prefilled with the patient, selected volumes, source, and destination, then Request creates Outbox rows and matching Inbox rows for custodians.

Creating a new request from Outbox

Open Outbox and choose New request. Select patient opens a search-like page. Select opens the volume selector for the chosen patient. Complete Required for, From, and To, then choose Request. Cancel returns without saving.

Using Outbox

Outbox lists requests made by the current user. It shows the patient with fbAddressograph, requested volumes with fbcntSelectedVolumes, request details, source, destination, status, and actions. Cancel cancels and hides a row. Locator opens the patient locator with those volumes selected. Done actions in Inbox update Outbox status to actioned.

Using Inbox

Inbox lists open requests for locations where the current user is a custodian. Send opens the same send workflow used by the locator. Done marks the request as actioned and removes it from Inbox.

Setting up a pick list

Open My clinics, choose Add clinic(s), filter future clinics by Health board, Locality, Facility, and Speciality, tick clinics, and choose Ok. The filter starts from the saved login preferences. My clinics then lists the clinic instances the user retrieves for.

Using a pick list

Open My picklist. Each row identifies a clinic, appointment, patient, and required volumes. Select opens Case notes selector for that patient and clinic instance; Ok saves the selected required volumes. Receive opens a popup for marking selected required volumes as received.

Using My return list

My return list looks like My picklist and lists patient volume groups that should be returned. Send opens the send workflow for the row volumes. Remove hides that row from the current return list.

Registering a volume

From Case notes locator choose Register volume. Enter Created date, Health board, Locality, Type, Status, optional permanent Volume number, Initial location, Bar code, and RfID. Register creates and selects the new volume; blank identifiers are generated safely.

Sending selected volumes

Select one or more volumes and choose Send selected. The popup collects From and To using CNT location selectors. Send updates each volume's current location, appends movement history, and saves the user's send-location preference.

Receiving selected volumes

Select one or more volumes and choose Receive selected. The popup infers a From location from a recent send where possible, requires To, and appends received movement history.

Managing batches

My batches lists favourite batches. Open shows the Batch page grouped by patient with addressographs and removable volume rows. Remove removes a favourite from the user's list. Find batch opens a filterable list of existing batches. Create batch captures Purpose, Current location, Intended destination, and Barcode.

Adding selected volumes to a batch

On Case notes locator choose Batch selected. The popup lists the user's favourite batches. Choosing one adds the selected volumes to that batch.

Using simulated batch scans

The home page Simulate batch number scan creates a fresh batch, adds a small sample of volumes, adds the batch to the user's favourites, and opens the Batch page.

Tagging volumes

On Case notes locator choose Tag selected. Enter Purpose and Location, then Create tag. Active tags display under volume location strings as an abbreviated single line; hovering or focusing the tag summary opens an fbToolTip with each tag field on its own line.

Managing libraries

My libraries lists favourite library locations and their custodians. Remove removes a library from the user's favourites. Add library asks for a location through fbcntLocation and stores it as a favourite.

Using identifier scanning

The CNT home scan bar accepts patient NHS numbers, hospital numbers, volume barcodes or RFIDs, batch barcodes, and location codes. Open resolves the identifier and opens the appropriate target or sets a send/receive destination for locations. Empty or unknown identifiers display an Error popup.

Data model overview

Table/queryPurpose
cnt_usersSimulated CNT identities, NADEX IDs, roles, specialities, and home facilities.
cnt_patientsFictitious patients used by the tracker, mirrored into patients for formBuilder2 search entry points.
cnt_locationsHealth board, locality, facility, department, request acceptance, and custodians.
cnt_volumesCurrent case-note volume state, identifiers, owning patient, created health board/locality, type, number, and current location.
cnt_volume_eventsCreated, sent, received, merged, and destroyed movement history.
cnt_batchesBatch barcode, current location, intended purpose, and intended destination.
cnt_batch_volumesMany-to-many membership between batches and volumes.
cnt_tagsActive or forgotten volume tags with purpose, required-by, expiry, location, and creator.
cnt_clinicsClinic definitions, speciality, facility, holding location, and clinician.
cnt_clinic_instancesDated clinic sessions with times, cancellation flag, and retrievers.
cnt_appointmentsPatient appointments within clinic instances.
cnt_requestsRetrieve requests from a source location to a destination location.
cnt_pick_listSelected pick-list volumes for a clinic instance and received status.
cnt_preferencesUser preferences, favourite batches/libraries, filters, and recent workflow choices.
search_patients_fuzzy(search_term)Main formBuilder2 fuzzy patient query used before opening CNT from Patient search.
cnt location searchFuture server query for fbcntLocation, matching health board, locality, facility, department, extra, and code.
cnt clinic instance searchFuture server query for Select clinic(s), filtering future non-cancelled instances.
cnt request inbox queryFuture server query for requests sent to locations where the user is a custodian.
cnt request outbox queryFuture server query for requests created by the user.

SQL query catalogue

-- Patient search before opening CNT from formBuilder2
SELECT * FROM search_patients_fuzzy(:search_term);

-- Locator volume tree
SELECT *
FROM cnt_volumes
WHERE patient_uuid = :patient_uuid
ORDER BY health_board, locality, type, temporary, volume_number;

-- Movement history
SELECT *
FROM cnt_volume_events
WHERE volume_uuid = :volume_uuid
ORDER BY event_datetime;

-- Location search for fbcntLocation
SELECT *
FROM cnt_locations
WHERE code ILIKE :query
   OR health_board ILIKE :query
   OR locality ILIKE :query
   OR facility ILIKE :query
   OR department ILIKE :query
   OR extra ILIKE :query
ORDER BY health_board, locality, facility, department;

-- Select clinic(s)
SELECT ci.*, c.*
FROM cnt_clinic_instances ci
JOIN cnt_clinics c ON c.uuid = ci.clinic_uuid
WHERE ci.clinic_date >= CURRENT_DATE
  AND ci.cancelled = false
  AND (:facility IS NULL OR c.facility = :facility)
  AND (:speciality IS NULL OR c.speciality = :speciality)
ORDER BY ci.clinic_date, ci.start_time;

-- Outbox
SELECT *
FROM cnt_requests
WHERE requested_by_user_uuid = :user_uuid
  AND status <> 'cancelled';

-- Inbox
SELECT r.*
FROM cnt_requests r
JOIN cnt_locations l ON l.uuid = r.from_location_uuid
WHERE :user_uuid = ANY(l.custodian_user_uuids)
  AND r.status = 'open';

Implementation notes

  • The current React prototype is client-side and stores runtime CNT state in localStorage.fbcntStore.
  • scripts/seed-cnt-real-db.ts seeds real PostgreSQL cnt_* tables from createInitialCntStore().
  • Local SWAS static deployment should be built with VITE_API_BASE_URL=/formBuilder2/api and copied to D:\swas\dev\root\formBuilder2.
  • Blocking workflow errors display in an fbPopup titled Error with a white-on-green Ok button.
  • Tables use fbTable, and table data cells are top-aligned.
  • CNT page body components are exposed as fbcntXxx files under src/caseNoteTracker; the top-level shell owns session, navigation, persistence, header/footer, and cross-page popups.
