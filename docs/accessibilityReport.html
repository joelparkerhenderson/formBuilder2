formBuilder2 accessibility report

This assessment covers the main patient-facing and clinician-facing app surfaces, excluding Composer. It is based on source inspection rather than a full manual assistive-technology test. The reference standard is WCAG 2.2 AA as described by the DWP Accessibility Manual: Web Content Accessibility Guidelines (WCAG).

Brief assessment

The app uses many native HTML controls, including buttons, inputs, selects, textareas, and tables. This is a good foundation for keyboard and screen-reader support. The visual design also uses strong colour contrast for the primary blue, green, red, black, and white interface.

The largest compliance risks are robust semantics and operability in custom widgets: popup modals, tooltips, autocomplete selectors, date pickers, history menus, and icon-only/status controls. Layout regions are often implemented as generic divs rather than landmarks. Form labels are visible but not always programmatically connected to inputs. These issues are likely to affect WCAG 2.2 AA criteria around name/role/value, info and relationships, keyboard operation, focus order, focus visible, and error/status communication.

Easy app-code changes made after the first report include a labelled form-section navigation landmark with `aria-current`, shared popup `role="dialog"` / `aria-modal` / `aria-labelledby` semantics, and additional accessible names for Composer/table icon actions touched during this pass.

Easy changes made

  • DoneShared `fbModal` now exposes `role="dialog"`, `aria-modal="true"`, and `aria-labelledby`.
  • Done`fbLayoutNav` is rendered inside `nav aria-label="Form sections"` where used by real edit forms and Composer.
  • DoneActive navigation buttons now set `aria-current`.
  • DoneComposer/table icon actions touched in this pass have explicit labels where they are icon-only.
